# 架构说明

公共版采用 monorepo 结构，把玩家前端、服务端、管理后台和共享协议包放在同一个干净仓库中。这样做的目标是让项目更容易本地启动、统一测试、统一发布，也避免公开版继续依赖私有 `shared` 仓库。

```text
apps/web         玩家网页客户端
apps/server      Node.js API 与 Colyseus 实时游戏服务端
apps/admin       独立管理后台
packages/shared  前后端共享协议与纯规则辅助函数
```

## 核心边界

项目的核心思路是把游戏规则、网络同步和 UI 渲染拆开：

- `packages/shared` 放前后端都要理解的类型、协议版本、动作结构和纯规则辅助函数。
- `apps/web` 负责界面渲染、按钮交互、动画表现和客户端路由。
- `apps/server` 负责联机房间、动作校验、权威状态、数据持久化和结果广播。
- `apps/admin` 负责运营和测试用的管理操作，不直接连接数据库。

这种拆分可以减少“单机逻辑”和“联机逻辑”互相缠绕的问题。前端只需要发送动作，服务端负责判断动作是否合法并广播结果。

## 单机与联机

单机模式可以由浏览器本地驱动规则层，适合快速试玩和离线体验。

联机模式应以服务端房间为唯一权威状态源。玩家点击出牌、摸牌、选择颜色等操作后，前端发送动作到服务端；服务端验证后更新状态，再同步给房间内所有玩家。

## 数据边界

账号、背包、邮件、好友、排行榜、兑换码、GM 操作等持久化系统都应通过服务端 API 访问。玩家前端和管理后台都不应该直接读写数据库。

推荐数据流：

```text
Web / Admin -> Server API -> Repository -> Database
Web 游戏房间 -> Colyseus Room -> Shared Rule Helpers
```

## 管理后台

这里的 Colyseus 指第三方实时多人游戏框架，本项目用它承载联机房间、WebSocket 通信和房间状态同步。

管理后台被拆成独立静态应用，方便单独部署、单独加访问控制，也方便未来扩展运营工具，例如：

- 玩家资料与背包管理。
- 邮件系统。
- 兑换码系统。
- 排行榜结算与奖励发放。
- 商品、称号、饰品等内容配置。

公开部署时，不建议把管理后台当普通页面暴露给所有人访问。

## 公共版差异

这份公共导出移除了私有部署细节，并使用占位域名：

- 玩家前端：`https://game.example.com`
- 服务端 API：`https://api.example.com`
- 管理后台：`https://admin.example.com`

实际部署时请替换为你自己的域名和环境变量。
